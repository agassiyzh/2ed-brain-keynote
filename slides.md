# 核心启蒙：一个简单的想法

<v-clicks>

## 不要每次都重新分析

**老方法**：遇到问题 → 搜旧邮件/Slack → 重新理一遍 → 费时间

**新方法**：遇到问题 → 查 Wiki → 已经整理好的答案 → 快速决策

### 怎样做到

1. **收集** — 把团队的发现记下来
2. **整理** — AI 帮你组织成 Wiki
3. **连接** — 相关的页面自动连起来
4. **保鲜** — 新信息来了，自动更新

### 结果

知识像一棵树一样生长，而不是一堆散乱的纸条

</v-clicks>

# 第一次对话：和 AI 讲清楚规则

## 你要写一份"指导手册"给 AI

<div class="text-sm">
<v-clicks>

### 你要说清楚

```
你的名字是 Wiki 助手。

你的工作：
把新文档整合到我们的 wiki 里

步骤：
1. 读新文档，找关键信息
2. 查看现有 wiki，找相关页面
3. 如果有相关页面就更新它
4. 如果没有就新建页面
5. 更新目录和相关链接

规则：
- 决策页面包含：问题、选项、选择、原因
- 坑点页面包含：问题、原因、解决方案、验证
- 工具页面包含：用途、优缺点、何时用
```

### 第一次设定好了，以后就可以重复使用

</v-clicks>

# 信心评分的力量

## 为什么我们要标记"哪些信息可信"

<div class="text-sm">
<v-clicks>

**场景**：用 A 还是 B 技术？

**没有评分时**：
- 查 Wiki → "推荐 A"
- 你不知道多可靠
- 可能是一年前的决策，早过时了

**有评分时**：
```
推荐 A（99% 确定）
- 最近检查：2 周前
- 5 个项目都在用
- 反面意见：B 也有好处（60% 确定）
- 权衡：A 更完整，值得
```

**结果**：
你知道这个决策有多稳定，什么时候可能改变

</v-clicks>
</div>

# 反向思考：这套系统不是"完美的"

<div class="text-sm leading-8">
<v-clicks>

**1. LLM 不完美** → Human-in-the-loop 审查、定期 lint

**3. 需要纪律** → 前期投入时间磨 Schema

**4. 不是"即插即用"** → 从小规模开始，逐步演进
